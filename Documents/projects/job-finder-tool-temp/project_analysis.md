# Job Hunter: Comprehensive Project Analysis & Developer Context

This document is designed to provide complete context of the **Job Hunter** repository. It covers the file structure, database schema, scraper implementations, anti-bot mechanisms, and typical data flow. 

---

## 1. Executive Summary
Job Hunter is an asynchronous, multi-source job aggregation tool. It is designed to run locally, fetching jobs from various platforms (Direct JSON feeds from ATS systems, HTML parsing from job boards, and automated browser scraping for authenticated/protected sites). The system standardizes these jobs, prevents duplicates via hashing, stores them in an SQLite database, and exposes the data via both a Rich CLI and a Streamlit dashboard.

## 3. End-to-End Data Workflow

### Phase 1: Extraction
The `run_scrapers.py` script orchestrates the collection process:
1. **Universal Aggregators (`universal.py`)**: Fires off scrapers for general job boards (Hacker News, RemoteOK, GitHub Jobs). These do not require company lists and fetch bulk data.
2. **ATS JSON Feeds (`feeds/`)**: Reads `app_config/companies.json`. For ATSs like Greenhouse or Lever, it directly queries their public JSON APIs (`https://boards-api.greenhouse.io/v1/boards/{company}/jobs`). This is blazing fast.
3. **Browser Scraping (`linkedin.py`, `browser/workday_browser.py`)**: Uses Playwright. LinkedIn specifically reads from a saved `session_data/linkedin_storage.json` file. If the session is missing, it alerts the user to run `login_linkedin.py` or set `DO_LINKEDIN_LOGIN = True` in the runner.

### Phase 2: Normalization & Deduplication
Every scraper yields a list of raw dictionaries representing a job.
1. Each job is passed through `parser.normalizer.normalize_job(job)`. It categorizes the employment type (Full-time vs Internship) based on keywords and cleans the text.
2. `parser.dedup.Deduplicator.generate_id(normalized)` generates a unique hash based on a combination of the `title`, `company`, and `location`.
3. The orchestrator checks this hash against a cached `Set` of existing hashes pulled from `db.get_existing_job_ids()`.

### Phase 3: Storage
If the job is entirely new, it is batched and inserted into the SQLite database via `database.db.insert_jobs(unique_jobs)`.

### Phase 4: Presentation
Data is never interacted with directly; it goes through one of two interfaces:
- **CLI**: `python -m interface.cli` (Uses `rich` tables and interactive inputs).
- **Web UI**: `python -m streamlit run interface/streamlit_app.py` (Visual dashboard with filtering).

## 4. Database Schema (`database/models.py`)

The application uses SQLite with **FTS5** (Full-Text Search) for extremely fast keyword querying.

### `jobs` Table
```sql
CREATE TABLE IF NOT EXISTS jobs (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    job_id TEXT UNIQUE NOT NULL,       -- Primary deduplication ID (generated by scraper or hashing)
    title TEXT NOT NULL,
    company TEXT NOT NULL,
    location TEXT,
    employment_type TEXT,
    description TEXT,
    requirements TEXT,
    apply_url TEXT NOT NULL,
    source_platform TEXT NOT NULL,     -- e.g., 'greenhouse', 'linkedin', 'hackernews'
    source_url TEXT,
    posted_date TEXT,
    scraped_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    is_active BOOLEAN DEFAULT 1,
    is_new BOOLEAN DEFAULT 1
);
```

### `jobs_fts` Virtual Table
Maintained by SQLite `AFTER INSERT/UPDATE/DELETE` triggers. 
```sql
CREATE VIRTUAL TABLE IF NOT EXISTS jobs_fts USING fts5(
    title, company, location, description,
    content='jobs', content_rowid='id'
);
```
Searches in `db.py` utilize this with the `MATCH` operator (`WHERE jobs_fts MATCH ?`).

## 5. The `playwright-stealth` Anti-Bot Implementation

A critical part of this application is bypassing anti-bot measures, particularly for LinkedIn and Workday. The project utilizes `playwright-stealth`. 

Because different versions of the `playwright-stealth` PyPI package expose different APIs (v1 exposes `stealth_async` directly, while v2 relies on a `Stealth().use_async()` context manager), the `scrapers/linkedin.py` implements a highly robust compatibility layer (`_stealth_context` and `_apply_stealth`).

```python
# The wrapper logic dynamically checks the stealth version:
async with async_playwright() as p:
    _sc = _stealth_context(p)  # Uses v2 ContextManager if available
    stealth_browser = await _sc.__aenter__()
    
    browser = stealth_browser if stealth_browser else await p.chromium.launch(headless=False)
    context = await browser.new_context(...)
    page = await context.new_page()
    
    await _apply_stealth(page) # Applies v1 per-page stealth if needed
```

## 6. Configuration Management (`app_config/companies.json`)

The ATS feed scrapers (Greenhouse, Lever, Ashby) rely on this JSON file. Because they query standard URLs (like `https://boards-api.greenhouse.io/v1/boards/{company}/jobs`), you only need the company's internal board identifier.

Example structure:
```json
{
  "greenhouse": [
    "google",
    "microsoft",
    "stripe",
    "vercel"
  ],
  "lever": [
    "raycast",
    "calcom"
  ],
  "unstop_queries": [
    "software engineer",
    "internship"
  ]
}
```

## 7. Key Dependencies
- `playwright`: Headless/headful browser automation for JS-heavy sites.
- `playwright-stealth`: Anti-bot evasion.
- `httpx`: High-performance asynchronous HTTP client.
- `selectolax`: Blazing fast HTML parsing (significantly faster than BeautifulSoup).
- `rich`: Terminal formatting and tables.
- `streamlit`: Dashboarding UI.
- `scrapy`: Though present in `requirements.txt`, the current iteration heavily favors `httpx` and `playwright`.
